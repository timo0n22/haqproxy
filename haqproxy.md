# ТЗ для Claude Code: hackproxy v2 (Go + htmx)

**Контекст:** есть рабочий прототип на Python (mitmproxy-аддон + Flask + vanilla JS) —
Proxy+History, Repeater, Scope. Он подтвердил архитектуру (проверено на реальном
трафике), но дальше переписываем на Go с более широким набором фич. Это
полноценный проект, не "пара скриптов" — рассчитывайте на несколько
итеративных сессий, а не один проход.

Аналог в голове держать — Caido, не Burp: лёгкий, быстрый, единый бинарник,
без Java-подобной тяжеловесности.

---

## 1. Верхнеуровневая архитектура

Два отдельных Go-бинарника из одного репозитория:

```
hackproxy-go/
├── cmd/
│   ├── hackproxy/        — бинарник для MacBook: proxy + вся история + фичи
│   └── collaborator/     — бинарник для VPS: DNS+HTTP OOB-слушатель
├── internal/
│   ├── proxy/             — MITM на базе goproxy
│   ├── store/               — SQLite (modernc.org/sqlite)
│   ├── replay/                — сырой сокет-клиент (бывший repeater.py)
│   ├── httpql/                  — мини-язык запросов для истории
│   ├── authmatrix/                — тестер доступа по ролям/сессиям
│   ├── scanner/                     — пассивный scanner-lite
│   ├── domlogger/                     — трекер DOM XSS через инъекцию скрипта
│   ├── collaboratorclient/              — генерация токенов + опрос VPS
│   └── web/                               — HTTP-хендлеры + html/template (htmx)
├── web/
│   ├── templates/                           — .html шаблоны (layout + фрагменты)
│   └── static/                                — htmx.js (вендор), css
├── collaborator/                                — VPS-часть: DNS+HTTP листенеры, отдельная SQLite
└── go.mod
```

## 2. Технологический стек — что берём и почему

| Задача | Библиотека | Почему |
|---|---|---|
| MITM-прокси | `github.com/lqqyt2423/go-mitmproxy` | Аддон-модель почти 1:1 с Python mitmproxy (`Requestheaders`/`Request`/`Responseheaders`/`Response`) — легче портировать логику из прототипа; из коробки WebSocket и HTTP/2; хук `StreamResponseModifier` для потоковой модификации тела ответа (нужен для DOMLogger-инъекции, см. §9). Активность репозитория и форков проверить перед стартом — не гарантирую свежесть на момент реализации |
| SQLite | `modernc.org/sqlite` | Чистый Go, **без CGO** — критично для лёгкой кросс-компиляции и единого статического бинарника (driver name в `database/sql` — `"sqlite"`, не `"sqlite3"`) |
| DNS-сервер (Collaborator) | `github.com/miekg/dns` | Стандарт де-факто для написания DNS-серверов на Go (CoreDNS и т.п. построены на нём) |
| Фронтенд | `html/template` + htmx (вендоренный `htmx.min.js` через `embed.FS`) | Сервер рендерит HTML-фрагменты, htmx подменяет их по месту — почти без своего JS, при этом полностью в одном статическом бинарнике, без Node/сборки |

## 3. Модель данных (SQLite, `internal/store`)

Переносим схему из Python-прототипа почти без изменений (уже проверена):

- `entries` (id, source['proxy'|'replay'], parent_id, timestamp, host, port, scheme, method, path, resp_status, duration_ms, raw_request, raw_response, error, note, tag)
- `scope` (id, pattern, enabled)

Добавляем для новых фич:

- `identities` (id, name, header_overrides — JSON-список пар заголовков/кук для подстановки, например Cookie/Authorization конкретного пользователя)
- `matrix_runs` (id, base_entry_id, created_at) + `matrix_results` (run_id, identity_id, status, body_len, body_hash, duration_ms) — результаты прогона одного запроса под разными identity
- `findings` (id, entry_id, rule_name, severity, detail) — результаты scanner-lite
- `oob_tokens` (token, note, created_at) — сгенерированные payload'ы для Collaborator (хранится на клиенте, VPS о них не знает заранее — просто логирует любой хит на `*.oob.<домен>`)

На VPS — отдельная, никак не связанная с основной, SQLite: `interactions` (id, kind['dns'|'http'], token, source_ip, timestamp, raw_query_or_request).

## 4. Proxy + History

Логика та же, что в Python-версии (перехват на `response`, запись сырых байт), но с одной важной оговоркой:

**Известный компромисс:** `go-mitmproxy` (как и любая Go-библиотека поверх `net/http`) представляет заголовки через `http.Header` — это map, а не список пар, поэтому оригинальный порядок и регистр заголовков **не сохраняются побайтово** при пассивном перехвате (в отличие от Python/mitmproxy, где мы читали `.fields` и получали точные сырые байты). Это ограничение экосистемы Go в целом, а не конкретно выбранной библиотеки — сменить библиотеку эту проблему не решит, нужен был бы полностью свой парсер поверх сырого `net.Conn`. Для подавляющего большинства задач это не критично — важно для request smuggling/WAF-байпасов, а там всё равно работаете через Replay с ручным сырым редактированием, где полный контроль есть. Если в будущем понадобится побайтовая точность и в пассивной истории — пишем свой HTTP-парсер поверх hijacked-соединения, но для v1 это осознанно не делаем ради скорости.

## 5. Replay (бывший Repeater)

Тот же принцип, что уже проверен и работает в Python (`repeater.py`) — **порт этой логики на Go, не изобретаем заново**:

1. Свой TCP/TLS-сокет (`net.Dial` + `tls.Client` с настраиваемым `InsecureSkipVerify`), не `net/http` — чтобы отправлять сырые байты запроса как есть, без нормализации заголовков.
2. Разбор ответа: читать до `\r\n\r\n`, распарсить статус-строку и заголовки, затем:
   - если есть `Content-Length` — дочитать ровно столько байт;
   - если `Transfer-Encoding: chunked` — читать до терминирующего `0\r\n\r\n`;
   - иначе — читать до закрытия соединения (с таймаутом).
3. Хранить `raw_response` целиком как получено, для отображения и записи в БД.

Эта логика уже проверена на реальном трафике (pypi.org, 44MB-ответ по Content-Length без обрыва) — переносите точно так же, никаких сюрпризов ожидать не должно.

## 6. HTTPQL-аналог (поиск по истории)

Простая грамматика v1 (не полная HTTPQL из Caido, минимально полезное подмножество):

```
condition := field.op:"value" | field.op:number
field     := host | method | path | status | body | source
op        := eq | cont (contains) | gt | gte | lt | lte   (gt/lt/gte/lte только для status)
expr      := condition (and|or condition)*
```

Примеры: `method.eq:"POST" and status.gte:400`, `host.cont:"api" and source.eq:"replay"`.

Реализация: простой токенайзер + компиляция в параметризованный SQL WHERE (никогда не подставлять значения напрямую в строку запроса — только через placeholder'ы, это наш собственный инструмент, но привычка та же, что учим на самих уязвимостях).

## 7. AuthMatrix-аналог (тестер доступа по ролям)

Суть: один и тот же запрос прогоняется под разными "личностями" (разные Cookie/Authorization), результаты — в сравнительную таблицу. Прямое попадание в вашу цель №1 по ROI — IDOR/broken access control.

- **Identities** — именованные наборы заголовков для подмены (например "Admin", "User B", "Unauthenticated"), создаются/редактируются в UI.
- **Запуск матрицы**: берём любой запрос из History или Replay-вкладки как базовый, выбираем identities, для каждой — подменяем нужные заголовки в сыром запросе и отправляем через тот же движок, что Replay.
- **Результат**: таблица identity × (status, длина тела, хэш тела, время). Простая эвристика для v1: если запрос под "низкопривилегированной" identity вернул тот же статус и близкую длину тела, что и baseline (например Admin) — подсвечиваем строку как потенциальный IDOR. Не нужно ничего умнее для старта — сложную дифф-логику добавим по необходимости.

## 8. Scanner-lite (пассивный)

Фиксированный набор правил v1 (не универсальный DSL — осознанно просто, под конкретные частые находки):

- Отсутствие security-заголовков (CSP, X-Frame-Options, HSTS, X-Content-Type-Options).
- Отражение значения параметра запроса в теле ответа без экранирования (грубая эвристика reflected XSS — не замена ручной проверки, а подсветка "стоит посмотреть").
- Признаки verbose-ошибок/стектрейсов в теле (regex на "Traceback (most recent", "at java.", "SQL syntax", "Exception in thread" и т.п.).
- JWT в заголовках с `"alg":"none"` или явно слабым `HS256` при наличии признаков подобранного секрета (последнее — просто пометка "проверить", не автоматический подбор).
- Упоминания путей вроде `.git/`, `.env`, `.bak` в ссылках/JS-бандлах.

Результаты — в `findings`, бейдж-счётчик в UI рядом с историей.

## 9. DOMLogger-аналог (DOM XSS sink tracker)

Настоящий DOMLogger++ — это браузерное расширение, которое хукает опасные DOM-синки
(`innerHTML`, `document.write`, `eval`, `location.href=` и т.п.) в контексте страницы.
Чтобы не тащить в проект ещё и браузерное расширение как отдельный технологический
стек — делаем проще и остаёмся в рамках одного бинарника:

- Для хостов **в scope** прокси инжектит маленький JS-сниппет в начало `<head>` HTML-ответов — через хук `StreamResponseModifier`, который в `go-mitmproxy` уже есть готовым (не нужно городить свой hijack ради этого, как пришлось бы с goproxy).
- Сниппет через `Object.defineProperty`/monkey-patching перехватывает вызовы опасных синков и шлёт `navigator.sendBeacon()` на локальный `/api/domlogger/report` (наш же Go-сервер, слушающий локально) с именем синка, усечённым значением и stack trace.
- Ограничение по сравнению с настоящим расширением: инжектированный скрипт не увидит вызовы синков, которые произошли ДО его выполнения (расширение исполняется в изолированном мире раньше) — но если инжектить первым тегом в `<head>`, это редкий случай на практике.
- Если в будущем понадобится точность настоящего расширения — это отдельная фича вне рамок этого ТЗ.

## 10. Collaborator (OOB) — клиент + VPS-сервер

### 10.1 Клиент (в составе `cmd/hackproxy`)
- Кнопка "Generate OOB payload" → случайный токен (12 символов, base32/hex), формирует `<token>.oob.<ваш-домен>`, сохраняет в `oob_tokens` с заметкой (например, "SSRF в параметре callback_url заказа #123").
- Экран "Interactions" — опрашивает VPS (`GET /api/interactions?since=<ts>` с `Authorization: Bearer <shared_secret>`), показывает все хиты (DNS/HTTP) с привязкой к вашим токенам.

### 10.2 Сервер (`cmd/collaborator`, разворачивается на VPS)
- **DNS-листенер** (`miekg/dns`) — авторитативно отвечает на зону `oob.<ваш-домен>` (см. ниже про делегирование), логирует КАЖДЫЙ запрос (полное имя, тип записи, IP резолвера, время) в свою SQLite, отвечает A-записью на IP самого VPS (чтобы последующий HTTP от той же цели тоже пришёл к нам).
- **HTTP(S)-листенер** — логирует любой входящий запрos на `*.oob.<домен>` (путь, заголовки, тело, IP), отвечает просто 200.
- **API для клиента** — `GET /api/interactions` с проверкой `Authorization: Bearer` (общий секрет в конфиге, не хардкодить в репозитории).

### 10.3 Инфраструктурный шаг (НЕ код, а настройка DNS у регистратора — сделать до того, как фича будет полезна)

**Важно: нужна именно NS-делегация поддомена, а не просто wildcard A-запись.**

- Если сделать просто `*.oob.cyberist.ru → A → <IP VPS>` — вы получите только HTTP-сигнал (если цель реально дошла до HTTP), но **не увидите сам факт DNS-резолва**. А DNS-резолв — самый надёжный сигнал: проходит почти через любой egress-firewall, который блокирует исходящий HTTP, но разрешает DNS.
- Правильно: `oob.cyberist.ru NS ns1.oob.cyberist.ru` + `ns1.oob.cyberist.ru A <IP VPS>` — тогда ваш собственный DNS-листенер на VPS реально получает и видит каждый запрос на любой случайный поддомен под зоной, а не только те, что вы заранее прописали.

**Подтверждено (скриншот панели Aeza):** в редакторе DNS-записей Aeza нет типа NS вообще — доступны только A, AAAA, TXT, CNAME, DNAME, ALIAS, MX, SRV. Точечно делегировать поддомен `oob` через штатный интерфейс нельзя. Рабочие варианты, по возрастанию усилий:

1. **Тикет в поддержку Aeza** — попросить добавить NS + glue A-запись для `oob.cyberist.ru` вручную. Не самоуправляемо, но многие хостинги это делают по запросу.
2. **Отдельный дешёвый домен только под Collaborator** (рекомендуется) — например `.xyz`/`.site`. Тогда не нужна точечная NS-делегация поддомена — просто указываете кастомные NS-серверы **для всего этого домена целиком** (свои, на VPS), а это универсально поддерживаемая операция у любого регистратора, в отличие от NS-записи внутри существующей зоны. Плюс отделяет тестовую инфраструктуру от основного домена.
3. **Перенос DNS-хостинга cyberist.ru на Cloudflare** (бесплатно) — через тумблер "собственные NS-серверы" в Aeza, указав NS Cloudflare. У Cloudflare NS-делегация поддоменов — штатная функция. Более комплексный шаг: нужно перенести туда и остальные существующие записи домена (MX и т.п.), прежде чем переключать NS, иначе они пропадут в момент переключения.

## 11. UI/IA на htmx (по духу Caido, не Burp)

Сайдбар-навигация, не жёсткая двухпанельная раскладка как раньше:

- **History** — таблица + строка HTTPQL-поиска сверху, обновление через `hx-trigger="load, every 4s"`.
- **Replay** — вкладки (состояние вкладок держим на сервере, в памяти процесса или в БД по session-id; переключение вкладки — `hx-get` на фрагмент вкладки).
- **Matrix** — экран AuthMatrix.
- **Findings** — список результатов scanner-lite.
- **Collaborator** — генерация токенов + список interactions.
- **Scope** — как было.

## 13. Что осознанно НЕ делаем в v1

- Полноценный plugin SDK (просто хардкодим фичи популярных плагинов, не generic-архитектуру расширений).
- Строгий RFC-парсинг chunked trailers/extensions (эвристика по терминатору, как в Python-версии, — этого достаточно).
- Настоящее браузерное расширение для DOM-синков (см. §9 — инъекция скрипта вместо этого).

**Не исключаем, а получаем бесплатно:** WebSocket-перехват — `go-mitmproxy` поддерживает его из коробки (`WebSocketStart`/`WebSocketMessage` хуки), так что можно логировать WS-сообщения в историю без дополнительных усилий, в отличие от изначального плана с goproxy, где это было сознательно отложено.
